1. Visão Geral

Este repositório existe para estudar um sistema real de bicicletas partilhadas. O objetivo não é só visualizar viagens: é perceber performance operacional, e-bike share, pressão sobre estações e perfis comportamentais de uso.

Em portefólio, este caso funciona muito bem porque combina dados públicos reais, problema operacional claro, modelação analítica e uma explicação honesta das limitações do dataset.
Dados reaisUsa trip history oficial e feeds GBFS do Citi Bike.
OLTPSQLite para armazenamento operacional e staging inicial.
OLAPDuckDB para fact tables, agregações e consumo analítico.
dbtModelos e testes para governança da camada analítica.
StreamlitDashboard para KPIs, e-bike share e top stations.
Machine LearningClassificação e clustering de perfis de viagem.

2. Problema de Negócio

O sistema de bicicletas partilhadas precisa de responder a três perguntas centrais:

  1. Quais estações estão em maior risco operacional?
  2. Quando e onde o uso de e-bikes cresce mais?
  3. Que grupos de uso existem nas viagens realizadas?
Neste caso falamos de perfis de viagem e não de perfis pessoais persistentes, porque o trip history público atual não expõe um identificador estável de utilizador.

7. OLTP - SQLite

OLTP aqui funciona como base operacional/staging local. Guarda:

  • trips
  • stations
  • station_status_snapshots
  • vehicle_types
  • system_information

Porque esta camada existe

  • uniformiza o acesso a dados de trip history e GBFS;
  • permite staging local com SQL relacional;
  • separa ingestão bruta da modelagem analítica.

Comando

python python/setup_sqlite.py --sample-rows-per-file 20000

8. OLAP - DuckDB

A camada analítica cria tabelas para perguntas de negócio e dashboards:

  • fct_trips
  • dim_stations
  • fct_station_status

Objetivo

  • medir duração de viagem;
  • medir e-bike share por hora;
  • medir estações com maior procura;
  • detetar risco de esgotamento/saturação.

Comando

python python/setup_duckdb.py

9. dbt - Governança

O projeto inclui uma camada dbt dedicada: 

dbt_citibike/

Função

  • staging dos dados base;
  • marts analíticos reutilizáveis;
  • testes de qualidade e relacionamentos.

Executar

cd dbt_citibike
cp profiles.example.yml profiles.yml
export DBT_PROFILES_DIR=.
dbt run
dbt test

Base teórica mínima

  • staging: normalização da origem.
  • marts: camada final de consumo.
  • tests: proteção contra quebras de qualidade.

10. Streamlit - Visualização

O dashboard mostra uma primeira visão do sistema:

  • total de viagens;
  • duração média;
  • percentagem de e-bike;
  • e-bike share por hora;
  • top start stations;
  • risco de utilização/saturação nas estações;
  • comparação de modelos para prever uso de e-bike;
  • importância das variáveis e perfis médios de clusters.

Como analisar no dashboard

BlocoO que observarDecisão analítica
E-bike por horaHoras com maior adoção de e-bike.Perceber quando a procura elétrica ganha peso.
Top estaçõesConcentração de partidas e duração média.Identificar pontos de maior procura operacional.
Risco de estaçõesUtilização e disponibilidade no snapshot GBFS.Priorizar análise de rebalancing.
Comparação de modelosF1, Accuracy, Precision, Recall e ROC AUC.Escolher modelo de e-bike sem olhar apenas para accuracy.
ClustersDuração, hora, membro/casual e proporção de e-bike por grupo.Traduzir perfis de viagem em narrativa operacional.

Comando

streamlit run streamlit/app.py

11. Machine Learning

Casos práticos incluídos

ProblemaTipoObjetivo
Uso de e-bikeClassificaçãoPrever se uma viagem tem maior probabilidade de usar e-bike.
Perfis de viagemClusteringEncontrar grupos comportamentais com base em duração, hora, dia e tipo de utilizador.

Comando

python python/ml_citibike.py

Base teórica mínima

  • Classificação: prever classe alvo, aqui e-bike vs não e-bike.
  • Clustering: agrupar viagens semelhantes sem rótulo.
  • Accuracy: % de acertos na classificação.
  • Precision: quando o modelo prevê e-bike, quantas vezes acerta.
  • Recall: de todas as viagens e-bike reais, quantas o modelo consegue apanhar.
  • F1: equilíbrio entre precision e recall.
  • ROC AUC: capacidade de separar e-bike de não e-bike ao longo de vários thresholds.

Entender, aplicar e analisar

EtapaO que fazerComo interpretar
Entender Separar problema de negócio: adoção de e-bike, procura por estação, duração e perfis de viagem. O modelo deve responder a uma pergunta operacional, não só produzir uma métrica.
Aplicar Correr python python/ml_citibike.py e abrir streamlit run streamlit/app.py. O dashboard compara Logistic Regression, Random Forest e Gradient Boosting para prever e-bike.
Analisar Comparar F1/Accuracy/ROC AUC, depois ler feature importance e clusters. Se uma métrica sobe mas a interpretação operacional piora, o modelo ainda precisa de revisão.

Experiências adicionais recomendadas

  • prever duração da viagem com regressão;
  • classificar member_casual para estudar padrões de utilizador;
  • fazer forecast de procura por hora ou por dia;
  • detetar estações anómalas com regras simples ou Isolation Forest;
  • validar modelos temporalmente, treinando em meses antigos e testando em meses recentes.

Dados dos últimos dois anos

python python/run_pipeline.py --years 2 --sample-rows-per-file 20000
python python/ml_citibike.py
streamlit run streamlit/app.py
Para manter o projeto leve, usa --sample-rows-per-file quando trabalhares com dois anos. Nesse modo, o script descarrega cada ZIP oficial, extrai apenas a amostra e apaga o ZIP logo de seguida. A página complementar ML, comparação, visualizações e dados de 2 anos explica os casos de teste, métricas e comandos para descarregar os últimos 24 meses oficiais.
Sem user ID persistente, este caso modela grupos de viagem, não grupos pessoais ao longo do tempo.
