1. O que este caso quer resolver

Este projeto usa dados públicos reais do Citi Bike para responder a perguntas operacionais e analíticas. Em vez de estudar só “viagens de bicicleta”, estudas um sistema urbano com pressão sobre estações, adoção de e-bikes, procura temporal e perfis de utilização.

Dados reaisTrip history oficial e snapshots GBFS do sistema.
OLTPSQLite como staging operacional local.
OLAPDuckDB para fact tables e análise agregada.
dbtGovernação da camada analítica com SQL e testes.
MLClassificação de uso de e-bike e clustering de perfis de viagem.
DashboardLeitura rápida de KPIs e estações relevantes.
Este caso é especialmente forte para portefólio porque mostra trabalho com dados públicos reais, ingestão, modelação analítica e interpretação de limitações do dataset.

2. Como usar o projeto

2.1 Preparar ambiente

python3 -m venv .venv
source .venv/bin/activate
python -m pip install --upgrade pip
pip install -r requirements.txt

2.2 Modo rápido com dados reais

python python/run_pipeline.py --months 1 --sample-rows-per-file 20000

Isto faz três passos:

  • download dos dados do Citi Bike;
  • carga para SQLite;
  • transformação para DuckDB.

2.3 Correr Machine Learning

python python/ml_citibike.py

2.4 Abrir dashboard

streamlit run streamlit/app.py

2.5 Validacao completa

python python/validate_project.py

3. Explicacao do codigo e da responsabilidade de cada ficheiro

Ficheiro O que faz Porque importa
python/download_citibike_data.py Descarrega archives de trip history e snapshots GBFS. É a porta de entrada dos dados reais. Sem esta camada, o resto do projeto não arranca.
python/setup_sqlite.py Cria a base SQLite e normaliza a ingestão dos CSVs e snapshots. Serve como staging local e resolve diferenças entre ficheiros fonte.
python/setup_duckdb.py Cria fct_trips, dim_stations e fct_station_status. Transforma dados crus em estruturas analíticas de leitura fácil.
dbt_citibike/models/ Aplica staging e marts governados por dbt. Ajuda a garantir qualidade e consistência da camada analítica.
python/ml_citibike.py Treina modelos para uso de e-bike e perfis de viagem. Transforma dados operacionais em hipóteses preditivas e segmentação.
streamlit/app.py Mostra KPIs e gráficos a partir do DuckDB. Permite validar rapidamente se a camada analítica faz sentido.

3.1 Como pensar o código em blocos

  • aquisição: buscar dados oficiais;
  • staging: pôr os dados em forma relacional;
  • modelação: criar entidades analíticas úteis;
  • consumo: dashboard e ML.

4. Teoria por tras do caso

4.1 Porque OLTP e OLAP aqui fazem sentido

Mesmo num estudo local, vale a pena separar OLTP e OLAP. A camada SQLite guarda granularidade operacional. A camada DuckDB já pensa em performance para análise e agregação. Isto reflete um padrão real de engenharia de dados.

4.2 Porque GBFS e trip history são diferentes

Trip history mostra viagens concluídas. GBFS mostra estado do sistema num instante. A combinação dos dois é importante: um diz-te o que aconteceu, o outro diz-te como o sistema estava.

4.3 O que o ML consegue e o que não consegue aqui

O projeto consegue prever propensão de uso de e-bike e agrupar viagens com comportamentos semelhantes. Mas há uma limitação importante: sem identificador estável de utilizador, estás a modelar perfis de viagem, não perfis pessoais persistentes.

5. Como interpretar os outputs

5.1 Bases de dados

  • data/processed/citibike_oltp.db: staging operacional.
  • data/processed/citibike_analytics.duckdb: camada analítica.

5.2 Outputs de ML

Ficheiro O que significa Como usar
citibike_ml_metrics.json Métricas principais da classificação e número de clusters. Serve para comparar iterações e saber se novas features melhoram o modelo.
ebike_feature_importance.csv Importância das variáveis para o uso de e-bike. Ajuda-te a perceber que sinais são mais influentes no problema.
trip_profile_clusters.csv Resumo dos grupos encontrados. Podes usar isso para descrever perfis de viagem e construir narrativa analítica.
Uma accuracy de cerca de 0.72 no uso de e-bike é um sinal interessante, mas o valor real está em perceber porque o modelo decide assim: hora, estação, tipo de utilizador, duração e contexto operacional.

6. O que mais podes analisar a seguir

Procura temporalHoras de pico, dias úteis vs fim de semana, sazonalidade mensal.
Performance de estaçõesEstações com maior pressão, maior uso e maior risco de saturação.
E-bike adoptionOnde o uso de e-bike cresce mais e em que contexto.
RebalancingQue estações parecem precisar de reposicionamento mais frequente.
GeografiaMapa de estações, clusters por zona e padrões urbanos.
ML avançadoForecasting por hora, modelos por estação e explicabilidade mais forte.

6.1 Perguntas analíticas fortes

  • Que estações combinam alta procura com baixa disponibilidade recorrente?
  • O uso de e-bike cresce mais em trajetos curtos ou longos?
  • Que horas do dia distinguem melhor membros e utilizadores casuais?
  • Que clusters parecem commuters e quais parecem lazer?

7. Como estudar este caso sem te perderes

  1. Corre o modo sample e confirma que a pipeline fecha.
  2. Abre o SQLite para ver as tabelas de staging.
  3. Abre o DuckDB para ver as fact tables.
  4. Lê os modelos dbt para perceber a camada governada.
  5. Corre o ML e abre os outputs gerados.
  6. Abre o Streamlit e tenta explicar cada gráfico com base nas tabelas.
Quando conseguires contar a história inteira — origem, staging, modelo, dashboard e interpretação — já não estás só a correr scripts. Já estás a dominar o caso.
